## 1. Executive Summary

Spawn Protocol is a Darwinian AI agent swarm deployed on Mantle mainnet.

The system runs multiple child agents in parallel. Each child manages an isolated wallet and makes yield decisions using Venice AI. The parent process evaluates every child on a risk-adjusted score. Underperforming children are recalled on-chain, their failure analysis is generated by Venice, pinned to IPFS, and recorded in a Mantle `LineageRegistry`. The replacement child inherits every ancestor post-mortem as prompt context, so each generation should avoid repeating prior failures.

The submission claim is:

1. AI agents execute real on-chain yield actions on Mantle.
2. Every spawn, recall, lineage write, and generation summary is verifiable on Mantle.
3. Each successor reads immutable ancestor failure memory.
4. Later generations should outperform earlier generations in risk-adjusted yield.

The currently deployed foundation is live and verified on Mantle. The remaining blockers before a live swarm run are operational, not contract-development blockers:

1. Fund the treasury with USDe and a small amount of MNT.
2. Smoke test Pinata/IPFS pinning.
3. Launch with live flags.
4. Keep the swarm running long enough to produce termination, respawn, and Gen 1 to Gen 3 comparison data.
5. Deploy the dashboard publicly and record the demo video.

## 5. What the System Does Right Now

### In Dry Run Mode

If live flags are not set:

```bash
node --env-file=.env --import agent/node_modules/tsx/dist/esm/index.cjs agent/src/parent.ts
```

The parent process:

1. Starts the local control server.
2. Creates/persists `swarm_state.json`.
3. Creates/persists `swarm_events.json`.
4. Derives deterministic child wallets.
5. Does not transfer real USDe.
6. Does not broadcast real spawn or recall transactions.
7. Forks child processes.
8. Lets children run with dry-run/simulated behavior where live writes are disabled.
9. Records local events suitable for dashboard development.

Dry run is safe and should be run before live mode after any runtime change.

### In Live Mode

The intended live command is:

```bash
ALLOW_LIVE_SPAWN=true \
ALLOW_LIVE_RECALL=true \
ALLOW_LIVE_CHILD_WRITES=true \
ALLOW_LIVE_GENERATION_POSTS=true \
node --env-file=.env --import agent/node_modules/tsx/dist/esm/index.cjs agent/src/parent.ts
```

With all flags enabled, the parent process:

1. Derives an isolated child wallet for each lineage/generation.
2. Transfers `$15` USDe from treasury to each child wallet.
3. Sends each child wallet a configurable MNT gas stipend.
4. Calls `SpawnFactory.spawnChild(lineageKey, generation, childWallet)`.
5. Forks a child process with only that child private key in its environment.
6. Evaluates each child every 75 seconds.
7. Calls Venice to summarize each generation and posts `GenerationResult` on-chain.
8. Terminates underperformers after two consecutive below-threshold evaluations.
9. Calls Venice to generate a post-mortem.
10. Pins the post-mortem JSON to IPFS using Pinata.
11. Calls `ChildAgent.recallChild(reason, cid)`.
12. Calls `LineageRegistry.pushCID(lineageKey, cid)`.
13. Respawns the next generation.
14. The replacement child fetches all ancestor CIDs and includes their constraints in its Venice prompt.

## 6. Architecture

### Smart Contracts

#### `ChildAgent.sol`

Role:

1. Stores `parent`, `wallet`, `active`, and `spawnTimestamp`.
2. Exposes `initialize(parent, wallet)` for clone initialization.
3. Exposes `recallChild(reason, ipfsCid)` for parent-authorized termination.
4. Emits `RecallChild(child, reason, ipfsCid, timestamp)`.

Important behavior:

1. `initialize()` can only be called once.
2. `_parent != address(0)` is required.
3. `_wallet != address(0)` is required.
4. Only `parent` can call `recallChild()`.
5. `recallChild()` sets `active=false`.
6. No function can set `active=true` after recall.
7. The contract does not custody USDe. The child EOA wallet holds funds and interacts with Aave.

#### `SpawnFactory.sol`

Role:

1. Deploys EIP-1167 clones of `ChildAgent`.
2. Initializes each clone with `msg.sender` as parent and the derived child wallet as `wallet`.
3. Attempts to register the clone in ERC-8004.
4. Emits `ChildSpawned(child, agentId, lineageKey, generation, timestamp)`.

Important behavior:

1. Constructor rejects zero child implementation.
2. Constructor rejects no-code child implementation.
3. Constructor rejects zero lineage registry.
4. `spawnChild()` rejects zero child wallet.
5. ERC-8004 registration is best-effort. If registry has no bytecode or registration reverts, spawn still succeeds with `agentId=0`.
6. `spawnChild()` is intentionally unrestricted. The caller pays its own gas and becomes parent for the spawned child. This is acceptable for the demo, but should be documented as a public factory design choice.

#### `LineageRegistry.sol`

Role:

1. Maintains append-only arrays of IPFS CIDs per `lineageKey`.
2. Emits `LineageUpdated(lineageKey, cid, generation, timestamp)` for post-mortem CID writes.
3. Emits `GenerationResult(lineageKey, summary, avgYieldBps, agentsTerminated, generation, timestamp)` for Venice-generated evaluation summaries.

Important behavior:

1. `owner` is set in constructor.
2. `allowedCallers[msg.sender] = true` in constructor.
3. `pushCID()` is restricted by `onlyAllowed`.
4. `postGenerationResult()` is restricted by `onlyAllowed`.
5. `allowCaller(address)` and `revokeCaller(address)` are owner-only.
6. `postGenerationResult()` is event-only and does not mutate lineage CID counts.
7. `getLineage()` returns the full CID array for a key.
8. `getLatestCID()` reverts if no lineage exists.

### TypeScript Agent Runtime

#### `parent.ts`

Role:

1. Orchestrates the full swarm.
2. Starts the control server.
3. Spawns children.
4. Funds child wallets.
5. Evaluates children.
6. Terminates and respawns underperformers.
7. Persists local state/events.

Key constants and environment variables:

| Name | Purpose |
|---|---|
| `SWARM_CHILD_COUNT` | Defaults to `5` |
| `CHILD_CYCLE_INTERVAL_MS` | Defaults to `30000` |
| `PARENT_EVALUATION_INTERVAL_MS` | Defaults to `75000` |
| `AAVE_USDE_BENCHMARK` | Defaults to `7.47` |
| `RISK_THRESHOLD` | Defaults to `0` |
| `CHILD_GAS_STIPEND_MNT` | Defaults to `0.05` |
| `CHILD_GAS_FUNDER_PRIVATE_KEY` | Optional gas funder; defaults to deployer key |

Child wallet derivation:

1. Deterministic derivation from treasury key, lineage key, and generation.
2. Restart-safe: same lineage/generation produces the same child wallet.
3. Recovery-safe: if parent crashes, the operator can derive the same child wallets and sweep funds.
4. Isolation-safe: child processes do not receive treasury or deployer private keys.

Funding path:

1. Treasury sends USDe to child wallet.
2. Deployer or child gas funder sends MNT to child wallet.
3. Child process uses only its isolated child private key for live Aave writes.

Termination path:

1. Parent sees two consecutive below-threshold evaluations.
2. Parent asks Venice for a post-mortem.
3. Parent pins JSON to IPFS.
4. Parent calls `recallChild()`.
5. Parent pushes CID to `LineageRegistry`.
6. Parent kills child process.
7. Parent spawns generation `n+1`.

#### `child.ts`

Role:

1. Runs one child agent loop.
2. Reads live Aave USDe yield.
3. Builds Venice prompt with ancestor context.
4. Gets decision from Venice.
5. Approves the Aave Pool for USDe on first supply if allowance is insufficient.
6. Executes Aave supply/withdraw if live writes are enabled.
7. Reports yield and positions to parent via IPC.

Important behavior:

1. Reads `CHILD_PRIVATE_KEY`.
2. Does not rely on `TREASURY_PRIVATE_KEY`.
3. Falls back to simulated values where live writes are disabled or reads fail.
4. Uses `buildAncestorContext()` so lineage memory is injected into successor prompts.

#### `venice.ts`

Role:

1. `executeYieldReasoning()` - market decision for each child.
2. `generatePostMortem()` - structured failure memory for terminated children.
3. `generateGenerationSummary()` - compact summary for on-chain `GenerationResult`.

Current API compatibility:

1. Venice billing is topped up.
2. Direct smoke test succeeded.
3. Explicit `enable_e2ee` was removed from request JSON because the API rejects it as an unrecognized top-level key.
4. Successful response metadata still reported `enable_e2ee=true`, so server-side E2EE appears enabled.

#### `lineage.ts`

Role:

1. `pushLineageCID()` writes post-mortem CIDs on-chain.
2. `postGenerationResult()` writes Venice summaries and metrics on-chain.
3. `getLineage()` reads all ancestor CIDs.
4. `getLatestLineageCID()` reads latest CID.
5. `buildAncestorContext()` fetches post-mortems from IPFS/local fallback and formats them for Venice prompts.

#### `ipfs.ts`

Role:

1. Pins JSON to Pinata.
2. Returns real IPFS CID if successful.
3. Throws if Pinata is unavailable.

Important demo caveat:

1. `parent.ts` catches Pinata failures and stores a `local:` fallback CID.
2. Local CIDs are useful for development.
3. Local CIDs are not acceptable for final judging because judges cannot fetch them.
4. Before live demo, run a Pinata smoke test and confirm a real `Qm...` or CIDv1 hash resolves through a public gateway.

### Dashboard

The dashboard exists and is intended to show:

1. Swarm overview.
2. Active/terminated/respawning agent cards.
3. Judge flow timeline.
4. Generation chart.
5. Termination events with IPFS and Mantlescan links.

Current public state:

1. Dashboard code exists.

3. `NEXT_PUBLIC_API_URL` must point to a public endpoint, not localhost.

## 8. Security and Fund Model

### Custody Model

1. `ChildAgent` contracts do not hold USDe.
2. Child EOAs hold USDe and interact with Aave.
3. Treasury funds child EOAs with USDe.
4. Deployer/gas-funder funds child EOAs with MNT for transaction gas.
5. Parent process controls orchestration and recall calls.

### Child Wallet Isolation

Before the fix, children inherited treasury private key access. That was unsafe. Current design:

1. Parent derives deterministic child keys.
2. Parent passes each child only its own `CHILD_PRIVATE_KEY`.
3. Parent removes `TREASURY_PRIVATE_KEY` and `DEPLOYER_PRIVATE_KEY` from child env.
4. If one child process is compromised, the treasury key is not exposed through that process.

### Fund Recovery

If parent crashes mid-run:

1. Read `swarm_state.json` for active lineage keys and generations.
2. For each active child, derive the same child wallet from treasury key, lineage key, and generation.
3. Sweep USDe/aUSDe/MNT manually back to treasury.
4. If child funds were supplied to Aave, first withdraw from Aave using the derived child key.

Data needed for recovery:

1. `TREASURY_PRIVATE_KEY`.
2. Lineage key.
3. Generation number.
4. Current strategy positions, if funds are in Aave.

### Current Maximum Loss Before Funding

Currently treasury has no USDe, so no USDe is at risk. After funding five children at `$15` each, the default capital at risk is:

```text
5 children * 15 USDe = 75 USDe
```

This should be recoverable if the treasury key and lineage/generation data are retained.

### Access Control Summary

| Function | Access |
|---|---|
| `SpawnFactory.spawnChild()` | Public by design; caller pays gas and becomes clone parent |
| `ChildAgent.initialize()` | Public but single-use; factory calls it immediately after cloning |
| `ChildAgent.recallChild()` | Only clone parent |
| `LineageRegistry.pushCID()` | `onlyAllowed` |
| `LineageRegistry.postGenerationResult()` | `onlyAllowed` |
| `LineageRegistry.allowCaller()` | Owner only |
| `LineageRegistry.revokeCaller()` | Owner only |

### Known Non-Blocking Risks

1. `spawnChild()` is unrestricted. This is acceptable for a public factory but should be documented.
2. ERC-8004 registry is not live. Events will show `agentId=0` until bytecode exists at the canonical address.
3. `getLineage()` returns a full array. Very large lineages can become expensive to read on-chain, but current demo lineage sizes are small.
4. `LineageRegistry` stores strings on-chain. This is acceptable for CIDs and short generation summaries, but do not post long prose.
5. If Pinata fails and local fallback CIDs are written on-chain, the judge-facing lineage evidence is weak. Smoke test Pinata before live demo.

## 15. Operational Troubleshooting

### Venice returns 402

Meaning:

1. Billing is not funded.

Fix:

1. Top up Venice.
2. Run the smoke test again.

### Venice returns 400 about `enable_e2ee`

Meaning:

1. Current API rejects explicit top-level `enable_e2ee`.

Fix:

1. Keep the explicit field removed from `venice.ts`.
2. Confirm successful response metadata still reports E2EE if needed.

### `fundChildWallet` says treasury USDe balance too low

Meaning:

1. Treasury was not funded with USDe.

Fix:

1. Swap MNT to USDe.
2. Send USDe to treasury address.
3. Recheck `balanceOf`.

### Child Aave writes fail

Likely causes:

1. Child wallet has no MNT.
2. USDe approval transaction failed.
3. Aave reserve state changed.
4. RPC or nonce issue.

Action:

1. Check child MNT balance.
2. Check the approval transaction, if one was submitted.
3. Check failed transaction on Mantlescan.
4. Stop live loop before repeated failures.

### Pinata fails and local CID appears

Meaning:

1. Post-mortem was not publicly pinned.

Fix:

1. Check `PINATA_JWT`.
2. Run Pinata smoke test.
3. Do not use local CIDs as final evidence.

### ERC-8004 agentId is zero

Meaning:

1. Canonical registry has no bytecode on Mantle.

Fix:

1. No fix required for demo.
2. `SpawnFactory` gracefully falls back.
3. When registry goes live, `agentId` should populate automatically.
